Here's the full plan, written so you can hand the actual clicking to Claude in Chrome. First, do one thing yourself before opening the extension: download eitnlr_song_finder_v2.html from our chat to your computer (Downloads folder is fine), because the extension will need the file on your machine to upload it to GitHub.

The overall plan

  1. Find the GitHub repo that holds the memovivo.co site code
  2. Add the file to the repo as find/index.html (this makes the URL memovivo.co/find — clean, permanent, no version number)
  3. Commit to the main branch
  4. Cloudflare Pages sees the commit and auto-deploys — usually live in under two minutes
  5. Verify by visiting memovivo.co/find and clicking through one full path

Step-by-step prompts for the Claude extension

Step 1 — Identify the repo and structure. Open github.com, log in, then tell the extension:

"Go to my GitHub repositories list and help me find the repo that contains the memovivo.co website. Once we find it, open it and tell me how it's structured — specifically whether the site files (like the eitn page) live in the repo root or inside a subfolder like 'public', 'site', or 'dist'. Don't change anything yet."

This matters because if your /eitn page lives at, say, public/eitn/index.html, then the new file needs to go to public/find/index.html — matching the existing pattern. The extension can confirm this by finding where the eitn page's file sits.

Step 2 — Upload the file. Once the location is confirmed:

"Navigate into [the correct folder] in this repo. Use the 'Add file' → 'Upload files' option. I'll drag in the file eitnlr_song_finder_v2.html from my Downloads folder — but before committing, I need it placed inside a new folder called 'find' and renamed to 'index.html'. If the upload interface won't let us rename or create a folder, use 'Add file' → 'Create new file' instead, type 'find/index.html' as the filename (GitHub creates the folder automatically when you type the slash), and I'll paste the file contents in. Commit directly to the main branch with the message 'Add family song finder tool'."

The "Create new file" fallback is the reliable path — GitHub's upload tool can't rename or create folders, but the create-new-file box makes the folder the moment you type find/ in the name. If pasting is needed, open the downloaded HTML file in Notepad (right-click → Open with → Notepad), Select All, Copy, and paste into GitHub's editor.

Step 3 — Watch the deploy. Then:

"Go to my Cloudflare dashboard, open the Pages project for memovivo.co, and check the Deployments tab. Confirm a new deployment triggered from the commit we just made and tell me when it shows as successful."

Step 4 — Verify live. Finally:

"Go to memovivo.co/find and confirm the page loads. Click through one complete path — choose 'A spouse or partner', then 'Just tell the truth about losing them', then any option — and confirm a result screen appears with song titles and their italic subtitles."

Two things that might come up

  • If the repo uses a build system (you'll know because the repo has files like package.json or a src folder rather than plain HTML sitting where the eitn page should be), the file may need to go into a specific static-assets folder instead. If Step 1 reveals that, tell me what the extension found and I'll give you the exact placement — don't guess, because a wrong placement just silently won't appear on the site.
  • Cloudflare permission prompts — the extension will ask your approval before committing or clicking anything consequential. That's expected; approve the commit when it shows you what it's about to do.

Once it's live, the last piece of housekeeping is deciding whether /find gets linked from your homepage or stays an unlisted URL you share only with directors — the discoverability question we talked about. That's a one-line decision you can make later; publishing it doesn't force the choice.
